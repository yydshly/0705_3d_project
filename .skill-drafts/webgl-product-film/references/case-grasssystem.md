# Case Notes: GrassSystemThreeJS

This case converted a procedural grass/soil Three.js demo into a learning and publishing project.

## What Was Useful

- The original project already had real WebGL value: GPU-instanced grass, shader-driven wind, procedural soil, moss, model loading, lighting, and post-processing.
- The transformation came from adding direction: a scene-building story, camera timeline, irreversible growth, wind changes, clouds, captions, voiceover, music, and MP4 packaging.
- The project became easier to understand after README entry points and a capability guide were added.

## Lessons

- A technical demo needs story and pacing before it feels like a product showcase.
- Camera rotation alone does not communicate capability.
- Grounding and scale problems are immediately visible in 3D product shots.
- Natural effects need irregular timing: wind should gust, grass should not swing uniformly, growth should not look like an object scaling up.
- Publishing quality depends as much on audio alignment, subtitles, cover image, and format as on the WebGL scene.

## Reusable Trigger

When a user says they want to understand a WebGL demo, improve the visual demo, record a shareable film, or turn a Three.js project into an immersive product showcase, apply the main workflow.
